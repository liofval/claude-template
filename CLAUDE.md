# コーディング原則

## 基本ルール（常に適用）

### DRY — 同じコードを繰り返すな
- 同じロジックが2箇所以上に現れたら、即座に共通関数やモジュールに切り出す。
- 新しいコードを書く前に、既存のユーティリティで解決できないか確認する。

### KISS — シンプルに保て
- 問題を正しく解決できる、最もシンプルな実装を選ぶ。
- ジュニアエンジニアが60秒で理解できなければ、シンプルにし直す。
- 読みやすさを犠牲にする「賢い一行コード」は書かない。

### YAGNI — 今必要ないものは作るな
- 今の要件だけを実装する。
- 「将来使うかも」という理由でパラメータ・フラグ・抽象化を追加しない。

## SOLID原則

### S — SRP：単一責任の原則
- 関数は1つのことだけを行う。
- ファイル／モジュールは1つの関心事だけを持つ。
- 「〜して、さらに〜する」と説明が必要なら、分割のサイン。

### O — OCP：開放閉鎖の原則
- クラスやモジュールは「拡張に対して開いており、修正に対して閉じている」べき。
- 動いている既存コードを書き換えるのではなく、拡張によって新しい振る舞いを追加する。
- 抽象（インターフェース・基底クラス・コールバック）を使って拡張ポイントを設ける。

### L — LSP：リスコフの置換原則
- サブクラスは、親クラスが使われている場所にそのまま置き換えられなければならない。
- 親クラスの保証を弱めたり、想定外のエラーを投げるオーバーライドは禁止。
- 置き換えてロジックが壊れるなら、継承モデルが間違っている——継承より合成を検討する。

### I — ISP：インターフェース分離の原則
- 使わないメソッドの実装をクラスに強制しない。
- 大きなインターフェースは、小さく目的を絞ったインターフェースに分割する。
- 汎用的な1つより、目的別の複数インターフェースの方が良い。

### D — DIP：依存性逆転の原則
- 上位モジュールは下位モジュールに依存しない——両者とも抽象に依存する。
- 具体的な実装クラスではなく、インターフェースや抽象型に依存する。
- 依存オブジェクトは内部でインスタンス化せず、外から注入する（コンストラクタ注入・DIコンテナ）。

---

### SoC — 関心の分離
- データ取得・ビジネスロジック・表示は別のレイヤーに分ける。
- I/O（入出力）と計算ロジックを同じ関数に混在させない。

---

## 命名

- 名前はコメントなしで意図が伝わること。
- 省略形は使わない（`uc` ではなく `userCount`）。
- 真偽値：`is`、`has`、`can` で始める（`isLoading`、`hasError`、`canSubmit`）。
- 関数：動詞＋名詞の形にする（`fetchUser`、`validateInput`、`buildPayload`）。
- 避ける命名：`data`、`temp`、`result`、`flag`、`x`
- 良い命名：`invoiceTotal`、`retryCount`、`isAuthenticated`

---

## コード品質

### 防御的プログラミング
- 外部からの入力を無条件に信用しない。必ずバリデーションしてから使う。
- null・undefined・空文字・想定外の型を明示的に処理する。
- 失敗しうる処理は、失敗を呼び出し元に必ず伝える（throw / エラー返却など）。

### 早すぎる最適化は諸悪の根源
- まず正しく動かす。最適化はプロファイリングでボトルネックを確認してから。
- 計測なき最適化は禁止。読みやすさを理由なく犠牲にしない。

### ボーイスカウトルール
- 触ったファイルは来たときより少しきれいにして去る。
- 明らかに不要なコードや命名の問題は、ついでに直す。
- 理由のない `TODO` コメントは残さない。必ずIssueやチケットと紐付ける。

### 車輪の再発明をしない
- 標準ライブラリや信頼できるライブラリで解決できないか先に調べる。
- コアでないロジックは、メンテナンスされている依存ライブラリを優先する。

---

## コメントとドキュメント

- コメントは **「なぜ（Why）」** を書く。**「何を（What）」** はコード自体が語る。
- 自明なコードにコメントは不要。
- 必要なケース：複雑なアルゴリズム、非自明な副作用、意図的な回避策。

```
// ✅ なぜかを説明する
// 上流APIがクライアントあたり100msのレート制限を設けているため遅延が必要
await sleep(100);

// ❌ 何をするかを説明する（コードと重複で無意味）
// 100ms待機する
await sleep(100);
```

## 作業フロー

1. **理解する** — コードを書く前に要件を正確に把握する。
2. **設計する** — 非自明な変更は、実装前にアプローチをまとめる。
3. **実装する** — 最もシンプルな正しい解を書く。
4. **確認する** — テストを実行するか、期待する動作を手動で確認する。
5. **整理する** — 命名の改善、重複の除去、ロジックの簡略化を行う。
6. **ステップ4を絶対に省略しない** — 動作確認なしにタスクを完了にしない。

---

## やってはいけないこと

- 1箇所でしか使わない抽象化を作らない。
- エラーを握りつぶさない（`catch (e) {}` は禁止）。
- 自分にしか読めないコードを書かない。
- 壊れた・未確認のコードをマージしない。
- 依存ライブラリを追加する前に、メンテナンス状況とライセンスを確認する。
